# 05 — Dependency Injection

## DI library: Injekt (Mihon fork)

**Confirmed: this fork still uses Injekt** — specifically the Mihon fork
`com.github.mihonapp:injekt`. There is no Hilt, no Dagger, no Koin.

File: `gradle/libs.versions.toml:42`

```toml
injekt = "com.github.mihonapp:injekt:91edab2317"
```

File: `app/build.gradle.kts:261`

```kotlin
implementation(libs.injekt)
```

The runtime API used in source is the standard `uy.kohesive.injekt.*` package
(Injekt was originally published under `uy.kohesive`). The Mihon fork adds a
Compose-related patch exposed via `dev.mihon.injekt.patchInjekt`, which is
called at the very top of `App.onCreate()` (see `03-entry-points.md`).

Injekt is a lightweight, reflection-free DI container: a global singleton
`Injekt` object holds a registry of factories keyed by type. It has no compile-
time validation and no scope hierarchy — everything is either a singleton or a
factory.

## How the DI graph is initialised

File: `app/src/main/java/eu/kanade/tachiyomi/App.kt:82-104`

```kotlin
override fun onCreate() {
    super<Application>.onCreate()
    patchInjekt()                                    // Compose patch (must run first)

    ...

    Injekt.importModule(PreferenceModule(this))      // 1. Preferences
    Injekt.importModule(AppModule(this))             // 2. Infrastructure
    Injekt.importModule(DomainModule())              // 3. Domain interactors + repos
    // SY -->
    Injekt.importModule(SYDomainModule())            // 4. SY extras
    // SY <--

    setupNotificationChannels()
    ...
}
```

Each `Injekt.importModule(module)` calls the module's
`InjektRegistrar.registerInjectables()` extension, which runs the
`addSingleton` / `addSingletonFactory` / `addFactory` registrations. **Order
matters**: a factory's `get<T>()` calls are resolved lazily at retrieval time,
but the registry must contain the binding by then, so modules that produce
dependencies must be imported before modules that consume them in their
*constructor* (rare here, since most factories use lazy `get()`).

The four modules:

| Module | File | Responsibility |
|---|---|---|
| `PreferenceModule` | `app/src/main/java/eu/kanade/tachiyomi/di/PreferenceModule.kt` | `PreferenceStore` + every `*Preferences` class |
| `AppModule` | `app/src/main/java/eu/kanade/tachiyomi/di/AppModule.kt` | Infrastructure singletons: SQLite drivers, `Database`/`AnimeDatabase`, caches, network, source managers, extension managers, download managers/provider/caches, tracker manager, JSON/XML/ProtoBuf, image saver, storage, local-source file systems, `ExternalIntents` |
| `DomainModule` | `app/src/main/java/eu/kanade/domain/DomainModule.kt` | ~200+ bindings: repository interface → impl (`addSingletonFactory<Repo> { RepoImpl(get()) }`) and every domain interactor (`addFactory { SomeInteractor(get(), get()) }`) |
| `SYDomainModule` | `app/src/main/java/eu/kanade/domain/SYDomainModule.kt` | TachiyomiSY-derived extra interactors (marked `// SY -->`) |

## Where DI bindings are declared — the registration API

Injekt modules implement `InjektModule` and override
`InjektRegistrar.registerInjectables()`. Inside that scope three registration
functions are available (all from `uy.kohesive.injekt.api`):

- `addSingleton(value)` — register an already-constructed instance.
- `addSingletonFactory<T> { … }` / `addSingletonFactory { … }` — lazy singleton;
  the lambda runs once on first `get<T>()`, the result is cached.
- `addFactory<T> { … }` / `addFactory { … }` — transient; the lambda runs on
  every `get<T>()` call.

`get<T>()` inside a factory lambda retrieves another registered binding
(lazily). Reified-type overloads mean `get()` (no explicit type) works when the
expected type is inferable.

### Example — `PreferenceModule` (small, all singletons)

File: `app/src/main/java/eu/kanade/tachiyomi/di/PreferenceModule.kt`


Note the **interface-to-impl** pattern for the few cases where there is an
abstraction:

```kotlin
addSingletonFactory<MangaDatabaseHandler> { AndroidMangaDatabaseHandler(get(), sqlDriverManga) }
addSingletonFactory<MangaSourceManager> { AndroidMangaSourceManager(app, get(), get()) }
```

The `<MangaDatabaseHandler>` type-parameter binds the interface; consumers
retrieve via `get<MangaDatabaseHandler>()` and Injekt returns the impl. This is
the closest Injekt gets to "scoping".

## How to ADD a new binding

There are two common patterns. Pick based on lifetime.

### Pattern A — Singleton service (one instance for the app process)

1. Decide which module owns it. Infrastructure → `AppModule`; preferences →
   `PreferenceModule`; a domain repository/interactor → `DomainModule`.
2. Add an `addSingletonFactory` line. If you have an interface, bind it with
   the explicit type parameter `<InterfaceType>`; otherwise the reified type
   is inferred from the lambda's return type.

```kotlin
// In AppModule.registerInjectables():
addSingletonFactory { MyNewService(app, get()) }
// or, with an interface:
addSingletonFactory<MyNewService> { MyNewServiceImpl(app, get()) }
```

3. Retrieve it via `Injekt.get<MyNewService>()` (eager) or
   `by injectLazy<MyNewService>()` (lazy delegate — preferred in long-lived
   objects like activities and ScreenModels).

### Pattern B — Transient interactor (new instance per `get()`)

```kotlin
// In DomainModule.registerInjectables():
addFactory { MyNewInteractor(get(), get()) }
```

Then `Injekt.get<MyNewInteractor>()` from inside a ScreenModel.

`TrackerManager(app)` is registered as a singleton in `AppModule.kt:209`:

and consumed lazily in `AppModule.kt:209`-adjacent code and elsewhere, e.g.


and retrieved inside `AnimeLibraryScreenModel` (and many others) via:


## Retrieving bindings — `Injekt.get<T>()` vs `injectLazy<T>()`

Two retrieval APIs are used throughout the codebase:

- **`Injekt.get<T>()`** — eager. Resolves the binding immediately. Used when
  you are already on a background thread or are constructing a one-shot
  dependency (e.g. `App.kt:152`: `MangaWidgetManager(Injekt.get(), Injekt.get())`).
- **`by injectLazy<T>()`** — lazy delegate. The binding is resolved on first
  field access and cached in the delegate. **This is the dominant pattern** in
  activities, Application, and ScreenModels. Imported from
  `uy.kohesive.injekt.injectLazy`.

  ```kotlin
  // App.kt:76-77
  private val basePreferences: BasePreferences by injectLazy()
  private val networkPreferences: NetworkPreferences by injectLazy()

  // MainActivity.kt:132-140
  private val libraryPreferences: LibraryPreferences by injectLazy()
  private val preferences: BasePreferences by injectLazy()
  private val animeDownloadCache: AnimeDownloadCache by injectLazy()
  private val downloadCache: MangaDownloadCache by injectLazy()
  private val chapterCache: ChapterCache by injectLazy()
  private val getAnimeIncognitoState: GetAnimeIncognitoState by injectLazy()
  private val getMangaIncognitoState: GetMangaIncognitoState by injectLazy()
  ```

Because `injectLazy` resolves at first access (not construction), it is safe to
put these delegates on a class that is instantiated *before* `App.onCreate()`
runs the `Injekt.importModule(...)` calls — as long as the field is not read
before `onCreate()` finishes. Activities, ScreenModels, and Voyager `Screen`s
all satisfy this because their fields are only read inside lifecycle callbacks
that fire after `Application.onCreate`.

A few call sites also use **eager `Injekt.get<T>()` inside a Composable** for a
one-shot read where a lazy field would be awkward — e.g.
`HomeScreen.kt:267-271`:

```kotlin
val pref = Injekt.get<LibraryPreferences>()
combine(pref.newAnimeUpdatesCount().changes(), pref.newMangaUpdatesCount().changes()) { … }
```

## Notes / gotchas

- **No scopes.** Everything is either a process-wide singleton or a factory.
  There is no `@ActivityScope`, `@ScreenScope`, etc. If you need per-screen
  state, that lives in a Voyager `ScreenModel` (via `rememberScreenModel { … }`),
  not in Injekt.
- **No compile-time validation.** A missing binding fails at *runtime* on the
  first `get<T>()` / `injectLazy` access, with a not-very-friendly Injekt
  exception. New bindings should be smoke-tested by navigating to the screen
  that uses them.
- **`patchInjekt()` must run first.** `dev.mihon.injekt.patchInjekt` (called at
  `App.kt:84`) patches the registry for Compose-friendliness; if you add code
  that triggers `injectLazy` resolution during `Application` construction
  (before `patchInjekt()` runs), you can crash. In practice all `injectLazy`
  fields are read inside `onCreate()` after the patch.
- **`get()` vs `get<T>()`.** Inside a factory lambda, `get()` (no type arg)
  works only when the target type is statically inferable from the constructor
  parameter. When it isn't, use the explicit form `get<SomeInterface>()` —
  e.g. `StoragePreferences(folderProvider = get<AndroidStorageFolderProvider>(), …)`
  in `PreferenceModule.kt:81-84`.
- **The "warm-up" block at the end of `AppModule`** (`AppModule.kt:230-242`)
  eagerly constructs the most expensive singletons on the main executor after
  `onCreate()` returns. When you add a new expensive singleton, consider adding
  it here too if it's likely to be needed by the first screen — this is purely
  a cold-start optimisation, not a correctness requirement.
- **Testing.** There is no Injekt test override layer in this fork (no
  `InjektTest` base). Tests that need fakes either construct the class under
  test directly with fake dependencies, or use `mockk` (see `libs.versions.toml`
  `mockk`).
